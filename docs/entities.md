# Entities & Flows

## 1. Overview

Сервис «Запись на звонок» — минималистичная система бронирования времени:

* Один владелец (owner)
* Гости без аккаунтов
* Динамическая генерация слотов
* Ключевое правило: отсутствие пересечений бронирований

---

## 2. Domain Entities

### 2.1 EventType

Тип события — шаблон встречи.

**Поля:**

* `id: UUID`
* `name: string`
* `description: string`
* `duration_minutes: int` (≥ 5, кратно 5)

**Назначение:**

* Определяет длительность встречи
* Используется для генерации слотов
* Используется при создании бронирования

---

### 2.2 Booking

Бронирование — факт записи на конкретный временной интервал.

**Поля:**

* `id: UUID`
* `event_type_id: UUID`
* `start_at: datetime (UTC)`
* `end_at: datetime (UTC)`
* `guest_name: string`
* `guest_email: string`
* `status: enum (confirmed | cancelled)`
* `created_at: datetime`

**Правила:**

* `end_at` вычисляется автоматически:
  `start_at + duration_minutes`
* Бронирования не могут пересекаться
* Отменённые бронирования освобождают слот

---

### 2.3 Slot (вычисляемая сущность)

Слот — доступный временной интервал для бронирования.

**Поля:**

* `start_at: datetime`
* `end_at: datetime`

**Особенности:**

* Не хранится в базе данных
* Генерируется динамически
* В API возвращаются только свободные слоты

---

## 3. Relationships

* Один EventType → много Booking
* Все Booking принадлежат одному владельцу
* Slot — производная от:

    * EventType.duration
    * диапазона времени
    * занятых Booking

---

## 4. Business Rules

### 4.1 Доступность времени

* Каждый день: **09:00 – 18:00**
* Фиксированная, не настраивается

---

### 4.2 Генерация слотов

Алгоритм:

1. Берётся диапазон дат (`from`, `to`)
2. Строится сетка времени (09:00–18:00)
3. Делится на интервалы по `duration_minutes`
4. Исключаются интервалы, пересекающиеся с Booking

---

### 4.3 Инвариант системы (ключевой)

> Запрещены пересекающиеся бронирования

Формально:

```
(start_at < existing.end_at) AND (end_at > existing.start_at)
```

---

### 4.4 Отмена

* Переводит Booking в `status = cancelled`
* Освобождает соответствующий слот

---

## 5. Flows

---

### 5.1 Guest Flow (основной сценарий)

#### Шаг 1. Получение типов событий

```
GET /event-types
```

Пользователь видит:

* название
* описание
* длительность

---

#### Шаг 2. Получение доступных слотов

```
GET /slots?event_type_id=...&from=...&to=...
```

Система:

* генерирует слоты
* исключает занятые
* возвращает только доступные

---

#### Шаг 3. Создание бронирования

```
POST /bookings
```

Вход:

* event_type_id
* start_at
* guest_name
* guest_email

Система:

* вычисляет `end_at`
* проверяет пересечения
* создаёт Booking

Ошибки:

* `409 Conflict` — слот занят
* `422 Validation` — некорректные данные

---

### 5.2 Owner Flow

#### Просмотр бронирований

```
GET /bookings
```

Особенности:

* один список для всех типов событий
* поддерживает фильтрацию:

    * по статусу
    * по диапазону дат
* поддерживает пагинацию

---

#### Управление типами событий

```
POST /event-types
PATCH /event-types/{id}
```

---

### 5.3 Отмена бронирования

```
PATCH /bookings/{id}/cancel
```

Система:

* меняет статус на `cancelled`
* делает слот снова доступным

---

## 6. Non-Goals (что НЕ входит)

* Авторизация и аккаунты
* Интеграции с календарями
* Уведомления (email/SMS)
* Настраиваемое расписание
* Временные блокировки слотов (hold)

---

## 7. Технические решения

* Все даты в UTC
* UUID для идентификаторов
* Слоты не хранятся (генерация на лету)
* Защита от пересечений:

    * на уровне базы данных (рекомендуется exclusion constraint)

---

## 8. Ключевая идея

Система intentionally упрощена:

* минимальный набор сущностей
* одна бизнес-инварианта
* отсутствие лишних состояний

Это делает её:

* предсказуемой
* простой в реализации
* удобной для расширения

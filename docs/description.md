## 1. Назначение

Сервис «Запись на звонок» — это упрощённая система бронирования времени (аналог Cal.com), позволяющая одному владельцу публиковать доступные временные слоты, а гостям — записываться на встречи без регистрации.

---

## 2. Архитектура

* Frontend: React SPA
* Backend: Laravel (REST API)
* API-first подход
* Формат данных: JSON
* Авторизация отсутствует

---

## 3. Роли

### Владелец (Owner)

* Единственный в системе (задан заранее)
* Управляет типами событий
* Просматривает список всех бронирований

### Гость (Guest)

* Не имеет аккаунта
* Выбирает слот и создает бронирование

---

## 4. Доменная модель

### EventType (тип события)

Шаблон встречи.

Поля:

* id (UUID)
* name
* description
* duration_minutes (int, кратно 5, ≥ 5)

---

### Booking (бронирование)

Факт записи на конкретное время.

Поля:

* id (UUID)
* event_type_id (UUID)
* start_at (datetime, UTC)
* end_at (datetime, UTC)
* guest_name
* guest_email
* status: confirmed | cancelled
* created_at

---

### Slot (слот, вычисляемый)

Временной интервал, доступный для бронирования.

Поля:

* start_at
* end_at

⚠️ В API возвращаются **только свободные слоты**

---

## 5. Бизнес-правила

### 5.1 Доступность

* Ежедневно: 09:00–18:00
* Фиксированная, не настраивается

---

### 5.2 Генерация слотов

* Слоты не хранятся в БД
* Генерируются динамически
* Зависят от `duration_minutes` выбранного EventType

---

### 5.3 Инвариант (ключевой)

Запрещены пересекающиеся бронирования:

Два Booking не могут пересекаться по времени независимо от типа события.

---

### 5.4 Создание бронирования

* Гость выбирает слот (start_at)
* `end_at` вычисляется автоматически:
  start_at + duration_minutes EventType
* При пересечении возвращается ошибка (409 Conflict)

---

### 5.5 Отмена бронирования

* Выполняется через отдельный endpoint
* Меняет status → cancelled
* Освобождает слот

---

### 5.6 Отсутствие защиты от гонок на уровне API

* Double booking предотвращается на уровне БД (например, exclusion constraint)
* API не реализует временные блокировки (hold)

---

## 6. API (обобщённо)

### Event Types

* GET /event-types
* POST /event-types
* PATCH /event-types/{id}

---

### Slots

* GET /slots

    * параметры: event_type_id, from, to
    * возвращает только доступные слоты

---

### Bookings

* POST /bookings
* GET /bookings (с пагинацией)
* PATCH /bookings/{id}/cancel

---

## 8. Технические ограничения

* Все даты в UTC
* UUID для идентификаторов
* Валидация:

    * email: корректный формат
    * duration_minutes: кратно 5 и ≥ 5

---

## 9. Границы системы

Сервис НЕ включает:

* авторизацию
* регистрацию пользователей
* интеграции с календарями
* уведомления (email, SMS)
* сложные правила доступности

---

## 10. Ключевая идея

Система минималистична:

* один владелец
* динамические слоты
* единственный инвариант — отсутствие пересечений

Это позволяет реализовать её быстро и без избыточной сложности.
